## Summary of concepts

- Security requirements should be considered early in a project, not only after implementation begins.
- Defining security expectations at the beginning helps shape architecture, trust boundaries, and later technical decisions.
- Rest is part of sustainable long-term progress.

---

## Deeper understanding (optional but recommended)

A useful takeaway from today:

**Security starts before code.**

When I define security requirements early, I am already influencing:
- trust assumptions
- access control ideas
- key management expectations
- communication model
- system boundaries

That is different from exploitation work, but still directly related to security thinking.
